Basic Queries for GO Annotation

Overview: Select Search to search for the string in the text box, or select Filters to apply the filters set. Select a check box or radio button to enable the associated search or filter.

This is the main interface for queries related to gene ontology. See GO Help for a description of terms. First, a few facts:

  1. A GO ID is only in the TCW database if it has an assigned or inherited hit.
  2. The Gene Ontology is constantly evolving, hence, TCW will only be consistent with the Gene Ontology if TCW was built with the latest GO database from www.geneontology.org (the GO version is listed in the Overview panel).

Search

  • GO ID: The GO number, e.g. GO:0003674, or 3674. (Must use one of these formats!).
  • GO Name: All GO names are searched for the substring.
Load File: The file should contain one GO ID or one GO Name per line, depending on whether GO ID or GO Name is selected. GO Name must be the exact text when loaded from file.

Filters

Seq-Hit
  • E-value: The best e-value is assigned to each GO. That is, the hits are associated with zero or more GOs, and the sequence-hit pairs have e-values; this is the best sequence-hit e-value associated with the GO.
  • #Seqs: By default, the number of sequences with at least one hit that is assigned the GO or a descendant GO of this GO. See #Seqs section towards the end, which explains how this will be different if the DE #Seqs is changed.
  • Evidence Codes (EvC): Check the box, then select Evidence Codes to bring up a panel of available evidence categories. The filter is passed if a GO has any of the selected categories. See Info for the mapping of Evidence Codes to Categories.
Level
See GO Help for a description of levels.
  • All Ontology: This toggles between "biological_process" (BP), "cellular_component" (CC) and "molecular_function" (MF).
  • Specific: Specify a specific level.
  • Range: Specify a range of levels.
  • Slim: If GO slims were identified when the database was created, this option will be present. If checked, only the GO slims will be shown (plus filtered by any other option, e.g. E-value). The type of GO slims is shown on the Overview panel, e.g. goslims_plant.
Enrich
If GO enrichment p-values have been computed, you will see a row labeled Enrich on the main window; otherwise, ignore this section. By default, the p-values are computed using GOseq package, see the Overview for the method.

Filter on the p-values as follows:

  1. Check the box to the left of Cutoff: and adjust the cutoff in the text entry box on the right.
  2. This is followed by for and a button. This button will initially be labeled Any N where N is the number of p-value columns. To change the selection, select the button which will bring up a panel of all p-value column names; select the columns to query on. Choose one of the two options for applying the cutoff, either:
    1. Any: show results for which any of the selected p-value columns passes the threshold,
    2. Every: show results for which all columns pass.
  3. #Seqs: see the #Seqs section towards the end of this Help.

Results

Select one or more filters, then click BUILD TABLE or ADD to TABLE to execute the search, which builds a new table or adds to the existing table, respectively.
Using the Columns button, columns can be added/removed. Columns can be rearranged by dragging (temporarily disabled) and sorted by clicking on a column heading.

Columns

  • These are all obvious except the column GO, which is the GO Ontology (BP, CC, MF).
  • P-values: See Decimal Display on upper left panel.
  • Evidence categories: The Long format shows the evidence codes found for the category, where the inherited codes are in parenthesis. The Short format just states 'Yes' if it has one or move evidence codes in the category.

Modify Table

  • Delete Selected and Deleted Unselected: The selected rows or unselected rows will be removed, respectively. The corresponding rows will be removed from the other (not-displayed) list.
  • Select All and Unselect All: Selects and unselects all rows, respectively.

  2. Table - the entire table

#Seqs

There are two occurrences #Seqs on this panel, and the second effects the column #Seqs.
  1. Seq-hit: #Seqs: If this is set, the GOs displayed must have at least this many sequences associated with the GO ID.
     
  2. Enrich: #Seqs: Briefly, if this is set to DE, the count is all sequences with the GO that are considered DE for the selected set of p-values. If it is upDE, it is the count of the sequences that are up-regulated, and dnDE is the count of the sequences that are down-regulated. The relation between the GOseq p-value and when a sequence is considered DE is explained below. NOTE: when this is it set to something other than All, it is much slower!
     
  3. Column #Seqs: This is typically all sequences that have a hit associated with the GO, unless the Enrich: #Seqs is changed from All.
     
  4. View Sequences: This shows the sequences that correspond with the #Seqs column in the table.
DE Sequence: Each sequence has the following annotations:
  1. Sequence length.
  2. Zero or more hits, where many are associated with GO IDs.
  3. Read counts for different conditions along with DE p-values between pairs of conditions.
GOseq uses all the above information to determine the over-represented GOs for two conditions. For input to GOseq, a binary vector is created of N sequences (where N is the number of sequences with GOs); the value for a sequence is a 1 if it is DE, else it is 0. The scientist that executes GOseq via runDE sets the cutoff for DE, e.g. 0.05. The Overview panel under "Processing" states what the cutoff was, e.g. RoSt:0.05 indicates that the corresponding DE p-values used a cutoff of 0.05. Hence, the DE count is the number that was used by GOseq to calculate the GO p-value, where DE=upDE+dnDE. TO BE CLEAR: The #DE sequences is not based on the cutoff from the panel, but based on the cutoff used for input to GOseq.

If the for filter is on two or more conditions where "and" is used (e.g. StRo and StOl), the #Seqs is an "or" of the individual conditions ( this is confusing, but its done that way to account for the sequences that went into the GO P-values).

DE-trim Algorithm

This is disabled in v3.1.8.

In order to help identify the GOs most worthy of investigation, a DE-based trimmed subset of the GO IDs is automatically computed, as follows:

  1. Find all GO IDs passing the specified DE filter.
  2. For all passed GOs, compute the product of their p-values ("combDE"). For failed GOs, set combDE is set to 2.
  3. Each GO ID is examined, along with its children in the GO hierarchy. If there is a child with better combDE score than the parent, the parent is removed; otherwise, the children are removed.
  4. The remaining, non-removed GO IDs are the DETrim subset.

Justification: Higher level nodes contain transcripts found in lower levels, i.e. a lower level has either a 'part-of' or 'is-a' relation to the higher level. Therefore, there is considerable redundancy in DE enrichment scores between different levels of the hierarchy and it makes sense to single out those GOs which appear to be primarily responsible for the enrichment in a given sub-tree.

  • If a parent (i.e. higher level node) has a better DE score than any of its children, that indicates that the differential effect is happening biologically at the level of the parent, and being inherited by those children, if any, which also are DE-enriched.
  • Conversely, if a parent has a worse DE score than one of its children, it indicates that the differential effect is happening biologically at the child level (or below).
